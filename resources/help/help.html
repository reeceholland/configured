CONFIGURED Help

CONFIGURED helps you create, validate, version, and export structured robotics configuration projects without editing parameter files by hand.

A project is saved as a .configured file and can contain project metadata, a configuration hierarchy, parameters, Git state, and export-ready values.

1. Start A Project

New Project

  • Select New Project from the home screen.
  • Enter the project name, version, platform, author, and company details.
  • Choose whether the project should be Git-managed.
  • Select a parent folder when creating a saved project.

Open Project

  • Select Open Project from the home screen.
  • Choose an existing .configured file.
  • The project opens in the editor with its saved metadata and tree.

Clone Remote

  • Select the remote connection option from the home screen.
  • Enter the repository URL and local parent folder.
  • After cloning, CONFIGURED opens the project file found in the repository.

2. Edit The Configuration Tree

The editor stores your robot configuration as a hierarchy: System, Subsystem, Component, and Parameter.

  • Add Child creates a child item beneath the selected item.
  • Remove Selected deletes the selected item from the tree.
  • The properties panel edits the selected item's name, type, description, and parameter fields.
  • Parameter items include key, value, unit, and required fields.
The tree shows validation state while you work. Fix highlighted fields before saving or exporting.

3. Validation

CONFIGURED validates both individual fields and the full project.

  • Project names cannot be empty or contain path separators.
  • Item names must use letters, numbers, underscores, or hyphens.
  • Required parameters must have a key and value.
  • Parameter keys must be unique across the project.
Invalid projects cannot be saved through the normal save workflow.

4. Save And Metadata

  • Save Project writes the current project to its .configured file.
  • Project Metadata edits project identity, version, platform, author, company, and Git-managed state.
  • The last-modified timestamp is updated when the project is saved.
  • Git-managed projects can store the latest commit hash in metadata.
Save before switching workflows so Git actions and exports use the latest project data.

5. Git Workflows

Git tools are available for saved projects that are marked as Git-managed. Long-running clone, pull, and push actions run in the background.

  • Git Status shows the current repository state.
  • Git Identity configures user name and email for commits.
  • Commit can save the project, stage changes, and create a commit.
  • Switch Branch changes branches after unsaved editor changes are resolved.
  • Pull checks local state before pulling remote changes.
  • Push sends commits to the configured remote, including first-push setup.
Save or commit local changes before switching branches or pulling remote changes.

6. Export Parameters

Use the export menu to write collected project parameters to external files.

JSON

Exports metadata and collected parameters to a readable JSON document.

XML

Exports metadata and collected parameters to an XML document.

Export metadata may include project name, version, Git-managed state, and Git commit hash.

7. Recommended Workflow

  1. Create, open, or clone a project.
  2. Edit the configuration tree and parameter values.
  3. Resolve validation errors.
  4. Save the project.
  5. Commit, pull, or push if the project is Git-managed.
  6. Export to JSON or XML when needed.

8. Troubleshooting

Git actions are unavailable

  • Make sure the project has been saved.
  • Make sure Git is installed and available in the system path.
  • Make sure the project folder is a valid Git repository.
  • Check that the project is marked as Git-managed in metadata.

Export fails

  • Make sure a project is open.
  • Choose a writable file location.
  • Resolve duplicate or empty parameter keys before exporting.

The project looks out of date

  • Save the project again.
  • Refresh Git status after commit, pull, or push operations.
  • For Git-managed projects, confirm the repository has at least one commit.
